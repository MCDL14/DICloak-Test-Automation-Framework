# Dicloak 自动化框架跨平台兼容改造计划书

更新时间：2026-06-15

## 一、结论

当前项目最适合采用“CDP 主链路跨平台 + 平台能力适配层 + 高风险能力分阶段降级”的方案。

不建议直接把现有代码按 Windows、macOS、Linux 到处加分支，也不建议马上重写为纯平台无关框架。当前框架的核心优势是通过 Playwright/CDP 控制 Dicloak Electron APP 渲染层，页面对象、定位 YAML、unittest 编排、恢复、截图、重试和日志体系已经成型。跨平台改造应该保留这条主链路，把 APP 生命周期、进程检测、系统代理、桌面弹窗、截图兜底、路径预检等 OS 相关能力收敛到 `core` 层。

目标形态：

```text
tests/p0/*              业务用例，尽量不感知平台
pages/*                 页面对象，继续以 CDP/DOM 操作为主
locators/*              定位 YAML，不引入平台分支
core/platform/*         平台识别、能力声明、APP/进程/代理/桌面适配
core/app.py             使用平台适配层启动、关闭、检测 APP
core/process.py         使用跨平台实现替代 Windows shell 优先逻辑
core/screenshot.py      继续优先 CDP，桌面截图按平台兜底
```

## 二、当前项目适配性评估

### 1. 已经适合跨平台的部分

- CDP 连接：`core/cdp_driver.py` 通过 `http://127.0.0.1:9222/json` 和 Playwright/WebSocket 连接 Electron 页面，只要 APP 启动时带 CDP 参数，三端都可复用。
- 页面对象：`pages/*` 大部分操作基于 DOM、Element Plus 结构、可见文本、表格行和弹窗，不依赖 Windows 原生控件。
- 定位体系：`locators/*.yaml` 是业务概念定位，天然可跨平台。
- unittest 编排：`run.py`、`core/runner.py`、`core/result.py` 不要求特定 OS。
- 截图主路径：`core/screenshot.py` 已经优先使用 CDP，失败后使用 `mss`，Windows UIAutomation 只是最后兜底。
- 文件读写：`core/files.py` 基本使用 `pathlib.Path`，具备跨平台基础。
- Streamlit UI：`ui/app.py` 和 `streamlit_runner.py` 属于 Python Web UI，三端可运行。

### 2. 需要重点改造的 Windows 绑定点

当前强绑定集中在以下位置：

- `core/app.py`
  - 使用 `exe_path`、`.exe`、`taskkill /F /T /IM`。
  - `process_name` 默认 `Dicloak.exe`。
- `core/process.py`
  - 使用 `tasklist`、PowerShell `Get-Process`、`wmic`、`Get-NetTCPConnection`。
  - 使用 `encoding="mbcs"`。
  - 进程命令行、PID、监听端口、可执行路径解析均是 Windows 实现。
- `core/ui_driver.py`
  - 依赖 `uiautomation`。
  - 处理 Windows 文件选择器、保存弹窗和 UIAutomation 截图。
- `core/windows_proxy.py`
  - 通过注册表和 WinINet 修改 Windows 系统代理。
- `tests/p0/proxy_management/test_01_create_custom_proxy.py`
  - 直接导入 Windows 系统代理工具。
- `tests/p0/environment_management/test_01_kernel_integrity.py`
  - 默认内核进程名 `GinsBrowser.exe`，依赖进程命令行和可执行路径。
- `tests/p0/global_settings/test_07_disable_packet_capture_software.py`
  - 使用 `taskkill` 关闭抓包工具进程。
- `requirements.txt`
  - `uiautomation` 当前无平台条件，macOS/Linux 安装会不合适。
- `config/config.example.yaml`
  - 当前主配置面向 Windows `.exe` 和 Windows 系统代理。

### 3. 模块跨平台风险分级

| 模块 | 当前用例数 | 风险等级 | 原因 |
| --- | ---: | --- | --- |
| 环境分组管理 | 6 | 低 | 主要是 APP 内部列表、弹窗、筛选、授权成员，基本走 CDP/DOM。 |
| 成员管理 | 15 | 中 | UI 用例多为 CDP/DOM；API 停用/到期停用用例涉及登录态恢复，但 OS 绑定较少。 |
| 代理管理 | 1 | 高 | 当前依赖 Windows 系统代理、外部代理服务连通性、代理检测结果。 |
| 环境管理 | 25 | 高 | 涉及 APP 内核进程、内核版本、打开/关闭环境、下载目录、导入导出和进程命令行解析。 |
| 全局设置 | 12 | 高 | 涉及内核浏览器行为、扩展管理、网站访问、抓包工具进程和文件上传/下载。 |
| 扩展管理 | 0 | 中高 | 目录已预留，后续大概率涉及本地扩展包上传、内核浏览器和文件路径。 |

## 三、备选方案对比

### 方案 A：全量原生三端适配

一次性实现 Windows、macOS、Linux 的 APP 启停、系统代理、桌面弹窗、进程检查、截图、文件选择和 CI 运行。

优点：

- 理论覆盖完整。
- 后续用例新增时不需要再补平台能力。

缺点：

- 改动面最大。
- macOS 权限、Linux 桌面环境和系统代理差异会拉长周期。
- 代理、抓包、内核进程类用例很难一次稳定。
- 容易把业务页面对象和平台细节搅在一起。

结论：不适合当前阶段。

### 方案 B：仅支持 `--attach-existing-app`

不管理 APP 生命周期，让用户在各平台手动启动 Dicloak 并带 CDP 参数，框架只连接已有 APP。

优点：

- 改动最小。
- 快速验证大部分 DOM 用例。
- 避开跨平台启动、关闭、进程检查。

缺点：

- 自动化闭环不完整。
- 预检、全量运行、CI 和失败恢复能力不足。
- 无法覆盖 APP 生命周期相关问题。

结论：适合作为第一阶段验证手段，但不能作为最终方案。

### 方案 C：渐进式平台适配层

保留 CDP 主链路，新增平台适配层。先跑通 attach 和基础模块，再逐步迁移 APP 生命周期、进程、截图、文件选择、系统代理和高风险用例。

优点：

- 贴合现有架构。
- 改动可分阶段提交。
- Windows 现有能力可保留，macOS/Linux 能力逐步补齐。
- 用例和页面对象保持业务语义，减少后续维护成本。
- 高风险模块可用能力判断和 skip 保护，避免非 Windows 上硬失败。

缺点：

- 需要维护平台能力矩阵。
- 早期 macOS/Linux 覆盖会分层，不会马上全量 59 条都绿。

结论：最适合当前项目。

## 四、推荐方案

采用方案 C：渐进式平台适配层。

核心原则：

1. CDP 是第一优先级，平台原生能力只做兜底。
2. 用例不直接判断 OS，统一通过平台能力层判断。
3. Windows 现有能力先平移进平台适配层，不做大重写。
4. macOS/Linux 先支持连接和低风险模块，再支持 APP 托管启动和高风险模块。
5. 系统代理、抓包工具、内核进程解析等能力允许阶段性 skip，不允许假通过。
6. 配置模板兼容旧字段，新字段逐步引导迁移。

## 五、目标架构

建议新增目录：

```text
core/platform/
  __init__.py
  detect.py
  capabilities.py
  app.py
  process.py
  proxy.py
  desktop.py
  windows.py
  macos.py
  linux.py
```

职责划分：

- `detect.py`：识别 `windows`、`macos`、`linux`。
- `capabilities.py`：返回当前平台能力，例如是否支持系统代理、桌面文件选择器、桌面截图、APP 托管启动、进程命令行读取。
- `app.py`：定义 APP 启动/关闭接口。
- `process.py`：定义进程查询、PID 检查、命令行读取、监听端口读取接口。
- `proxy.py`：定义系统代理启停、读取和恢复接口。
- `desktop.py`：定义文件选择器、保存弹窗、桌面截图接口。
- `windows.py`：迁移现有 Windows 实现。
- `macos.py`：实现 macOS 启动、进程和可选系统代理。
- `linux.py`：实现 Linux 启动、进程和可选系统代理。

示意接口：

```python
@dataclass
class PlatformCapabilities:
    app_managed_launch: bool
    process_command_line: bool
    process_listening_ports: bool
    desktop_file_dialog: bool
    desktop_screenshot: bool
    system_proxy: bool


class ProcessAdapter:
    def is_process_running(self, process_name: str) -> bool: ...
    def is_pid_running(self, pid: int) -> bool: ...
    def process_command_line_by_pid(self, pid: int) -> str: ...
    def process_executable_path_by_pid(self, pid: int) -> str: ...
    def listening_ports_by_pid(self, pid: int) -> list[int]: ...


class SystemProxyAdapter:
    def proxy_server_from_config(self, config: dict) -> str: ...
    def enable(self, proxy_server: str): ...
    def disable(self): ...
    def read(self): ...
```

## 六、配置演进方案

保留旧配置：

```yaml
app:
  exe_path: "C:/Program Files/Dicloak/Dicloak.exe"
  work_dir: "C:/Program Files/Dicloak"
  process_name: "DICloak.exe"

windows_system_proxy:
  host: "127.0.0.1"
  port: "7897"
```

新增跨平台配置：

```yaml
platform:
  name: auto

app:
  launch_mode: managed
  startup_args:
    - "--remote-debugging-port=9222"
    - "--remote-allow-origins=*"

platforms:
  windows:
    executable_path: "C:/Program Files/Dicloak/Dicloak.exe"
    work_dir: "C:/Program Files/Dicloak"
    process_name: "DICloak.exe"
    browser_process_name: "GinsBrowser.exe"

  macos:
    app_path: "/Applications/DICloak.app"
    executable_path: "/Applications/DICloak.app/Contents/MacOS/DICloak"
    work_dir: "/Applications/DICloak.app/Contents/MacOS"
    process_name: "DICloak"
    browser_process_name: "GinsBrowser"

  linux:
    executable_path: "/opt/dicloak/dicloak"
    work_dir: "/opt/dicloak"
    process_name: "dicloak"
    browser_process_name: "ginsbrowser"

system_proxy:
  host: "127.0.0.1"
  port: "7897"
  macos_network_service: "Wi-Fi"
  linux_backend: "gnome"
```

配置兼容规则：

- Windows 上优先读取 `platforms.windows.*`，缺失时回退到现有 `app.exe_path`、`app.work_dir`、`app.process_name`。
- `system_proxy` 缺失时回退到 `windows_system_proxy`。
- `config/test_data.yaml` 继续只放测试数据，不放平台运行能力。
- 本地真实配置仍由 `.gitignore` 排除。

## 八、用例能力策略

跨平台后用例状态分三类：

- `supported`：当前平台完整支持，正常执行。
- `degraded`：当前平台支持主要业务断言，但跳过某个系统兜底或非关键检查。
- `skipped`：当前平台缺少必要能力，明确跳过并输出原因。

示例：

```python
caps = platform_capabilities()
if not caps.system_proxy:
    self.skipTest("system proxy is not supported on this platform")
```

不允许：

- 因平台能力缺失而直接让用例失败。
- 因平台能力缺失而删除断言假装通过。
- 在测试文件中散落大量 `platform.system()` 判断。

## 九、验收标准

### Windows 回归标准

- 原有 59 条 P0 用例发现数量不下降。
- `python -m compileall -q core pages tests` 通过。
- `git diff --check` 通过。
- 低风险模块目录运行通过。
- 高风险模块若失败，必须明确是业务环境、外部代理或真实产品行为，不是跨平台重构引入。

### macOS 第一阶段标准

- 能安装依赖。
- 能用 `--attach-existing-app` 连接已启动并带 CDP 参数的 Dicloak。
- 能运行环境分组管理模块。
- CDP 截图可用；桌面截图若受权限影响，日志给出明确原因。
- 不支持的系统代理/桌面弹窗用例明确 skip。

### Linux 第一阶段标准

- 能在带图形会话的 Linux 桌面或 Xvfb 环境启动 Python 框架。
- 能用 `--attach-existing-app` 连接已启动并带 CDP 参数的 Dicloak。
- 能运行环境分组管理模块。
- 不支持的系统代理/桌面弹窗用例明确 skip。

## 十、主要风险和对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| macOS `.app` 参数传递不稳定 | CDP 无法打开 | 优先执行 `.app/Contents/MacOS/<binary>`，不用 `open -a` 作为主路径。 |
| Linux Wayland 截图/窗口控制受限 | 桌面兜底失败 | 优先 CDP 截图；Linux 文档建议 X11 或 Xvfb。 |
| 系统代理三端行为差异 | 代理用例不稳定 | 分阶段支持；非支持平台 skip；优先验证业务 UI 创建/删除。 |
| 内核浏览器进程名不同 | 环境管理内核用例失败 | 配置化 `browser_process_name`，进程接口读取命令行和端口。 |
| 文件选择器差异 | 导入导出用例失败 | 优先 DOM input 和下载事件，桌面弹窗只做兜底。 |
| 平台分支污染测试代码 | 后续维护困难 | 所有平台判断收敛到 `core/platform`。 |
| 本地真实配置泄露 | 安全风险 | 保持 `config/config.yaml`、`config/test_data.yaml` ignored；模板只放空值和示例。 |

## 十一、建议里程碑

### M1：可连接

- 平台识别和能力层完成。
- macOS/Linux 可 `--attach-existing-app` 跑通低风险模块。

### M2：可托管

- APP 生命周期跨平台完成。
- 三端可由框架启动和关闭 APP。

### M3：可诊断

- 进程命令行、监听端口、截图兜底完成。
- 内核运行时定位在三端可用。

### M4：可分层回归

- 低/中风险模块跨平台稳定。
- 高风险模块通过能力矩阵明确支持、降级或跳过。

### M5：可全量推进

- Windows 保持全量能力。
- macOS/Linux 达到约定模块覆盖。
- 系统代理和抓包工具类用例完成平台实现或正式标记限制。
