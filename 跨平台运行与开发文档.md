# Dicloak 自动化框架跨平台运行与开发文档

更新时间：2026-06-15

本文档配套 `跨平台兼容改造计划书.md` 使用，记录后续在 Windows、macOS、Linux 上运行和开发自动化框架时应遵守的规则。

## 一、适用范围

本文档适用于：

- 本地运行 Dicloak Electron APP 自动化。
- 使用 CLI 或 Streamlit UI 触发用例。
- 在 macOS/Linux 上验证框架跨平台能力。
- 后续开发新的页面对象、用例、平台适配器和配置模板。

当前框架仍以 Windows 为完整能力平台；macOS/Linux 按阶段支持。

## 二、推荐运行模式

跨平台改造期间，三端推荐先使用 `--attach-existing-app` 运行。

原因：

- 可以先验证 CDP、页面对象和用例逻辑。
- 避开 APP 托管启动和进程关闭差异。
- 更容易定位问题属于框架、APP、配置还是平台能力。

通用命令：

```bash
python run.py --config config/config.yaml --module environment_group_management --attach-existing-app
python run.py --config config/config.yaml --module member_management --attach-existing-app
python run.py --config config/config.yaml --level P0 --attach-existing-app
```

APP 必须以 CDP 参数启动：

```bash
--remote-debugging-port=9222
--remote-allow-origins=*
```

## 三、Windows 运行说明

Windows 是当前完整能力平台。

安装：

```powershell
pip install -r requirements.txt
playwright install chromium
```

运行：

```powershell
python run.py --config config/config.yaml --precheck
python run.py --config config/config.yaml --module environment_group_management
python run.py --config config/config.yaml --module proxy_management --attach-existing-app
```

当前 Windows 支持：

- 框架托管启动 Dicloak APP。
- `taskkill` 关闭 APP 或工具进程。
- 通过 CDP/Playwright 操作 APP。
- 通过 Windows UIAutomation 处理文件选择器、保存弹窗和截图兜底。
- 通过注册表和 WinINet 启停系统代理。
- 读取内核进程命令行、可执行路径和监听端口。

注意：

- 真实 `config/config.yaml` 和 `config/test_data.yaml` 不提交。
- 代理账号、成员 API token、飞书 webhook 只能写入 ignored 的本地配置或环境变量。
- 代理管理用例依赖外部代理服务连通性，失败时要区分框架问题和代理服务问题。

## 四、macOS 运行说明

### 1. 依赖安装

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
playwright install chromium
```

后续依赖拆分后，macOS 不应安装 Windows 专属的 `uiautomation`。

### 2. 手动启动 APP

优先直接执行 `.app` 内二进制，参数传递更稳定：

```bash
"/Applications/DICloak.app/Contents/MacOS/DICloak" \
  --remote-debugging-port=9222 \
  --remote-allow-origins=*
```

如果必须使用 `open`：

```bash
open -a "/Applications/DICloak.app" --args \
  --remote-debugging-port=9222 \
  --remote-allow-origins=*
```

推荐先验证 CDP：

```bash
curl http://127.0.0.1:9222/json/version
```

注意：`/json/version` 可访问只说明 CDP HTTP 入口存在，不代表 Playwright 已完成 browser-level attach。若日志停在 `ws connected` 后 `BrowserType.connect_over_cdp` 超时，优先重启 DICloak、关闭 DevTools 和残留子进程。框架页面对象只支持 Playwright CDP；raw websocket 不作为页面自动化兜底。

### 3. 运行用例

```bash
python run.py --config config/config.yaml --module environment_group_management --attach-existing-app
```

### 4. macOS 权限

如果后续启用桌面截图或原生文件选择器兜底，可能需要：

- Screen Recording
- Accessibility

第一阶段不强依赖这些权限，优先走 CDP 截图和 DOM 文件上传。

### 5. 系统代理

macOS 系统代理后续通过 `networksetup` 实现。配置示例：

```yaml
system_proxy:
  host: "127.0.0.1"
  port: "7897"
  macos_network_service: "Wi-Fi"
```

示例命令：

```bash
networksetup -setwebproxy "Wi-Fi" 127.0.0.1 7897
networksetup -setsecurewebproxy "Wi-Fi" 127.0.0.1 7897
networksetup -setwebproxystate "Wi-Fi" on
networksetup -setsecurewebproxystate "Wi-Fi" on
```

关闭：

```bash
networksetup -setwebproxystate "Wi-Fi" off
networksetup -setsecurewebproxystate "Wi-Fi" off
```

未实现前，代理管理系统代理用例应 skip。

## 五、Linux 运行说明

### 1. 依赖安装

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
playwright install chromium
```

如需 Playwright 系统依赖：

```bash
playwright install-deps chromium
```

### 2. 图形环境

推荐优先使用有桌面的 Linux 环境验证。CI 或服务器可考虑 Xvfb：

```bash
xvfb-run -a python run.py --config config/config.yaml --module environment_group_management --attach-existing-app
```

Wayland 下桌面截图、窗口控制、文件选择器兜底可能受限制。第一阶段优先 CDP，不依赖桌面控制。

### 3. 手动启动 APP

按实际安装路径启动 binary 或 AppImage：

```bash
/opt/dicloak/dicloak \
  --remote-debugging-port=9222 \
  --remote-allow-origins=*
```

验证 CDP：

```bash
curl http://127.0.0.1:9222/json/version
```

运行：

```bash
python run.py --config config/config.yaml --module environment_group_management --attach-existing-app
```

### 4. 系统代理

Linux 系统代理分桌面环境。后续第一目标只支持 GNOME `gsettings`：

```bash
gsettings set org.gnome.system.proxy mode 'manual'
gsettings set org.gnome.system.proxy.http host '127.0.0.1'
gsettings set org.gnome.system.proxy.http port 7897
gsettings set org.gnome.system.proxy.https host '127.0.0.1'
gsettings set org.gnome.system.proxy.https port 7897
```

关闭：

```bash
gsettings set org.gnome.system.proxy mode 'none'
```

KDE、无桌面环境、容器环境暂不承诺系统代理能力。未实现时相关用例应 skip。

## 六、配置规范

跨平台配置建议逐步迁移到以下结构：

```yaml
platform:
  name: auto

app:
  launch_mode: managed
  startup_args:
    - "--remote-debugging-port=9222"
    - "--remote-allow-origins=*"

platforms:
  windows:
    executable_path: "C:/Program Files/Dicloak/Dicloak.exe"
    work_dir: "C:/Program Files/Dicloak"
    process_name: "DICloak.exe"
    browser_process_name: "GinsBrowser.exe"

  macos:
    app_path: "/Applications/DICloak.app"
    executable_path: "/Applications/DICloak.app/Contents/MacOS/DICloak"
    work_dir: "/Applications/DICloak.app/Contents/MacOS"
    process_name: "DICloak"
    browser_process_name: "GinsBrowser"

  linux:
    executable_path: "/opt/dicloak/dicloak"
    work_dir: "/opt/dicloak"
    process_name: "dicloak"
    browser_process_name: "ginsbrowser"

system_proxy:
  host: "127.0.0.1"
  port: "7897"
  macos_network_service: "Wi-Fi"
  linux_backend: "gnome"
```

兼容要求：

- 旧字段 `app.exe_path`、`app.work_dir`、`app.process_name` 暂时保留。
- 旧字段 `windows_system_proxy` 暂时保留。
- 新代码读取配置时应先读新结构，缺失时回退旧结构。
- 测试数据继续放在 `config/test_data.yaml`，不要混入平台运行能力。

## 七、开发规范

### 1. 平台判断集中到 core 层

不要在测试用例里直接写：

```python
if platform.system() == "Windows":
    ...
```

推荐：

```python
from core.platform import capabilities

caps = capabilities.current()
if not caps.system_proxy:
    self.skipTest("system proxy is not supported on this platform")
```

### 2. 页面对象保持业务语义

页面对象方法应继续表达业务动作：

```python
proxy_page.open_list()
proxy_page.open_create_dialog()
proxy_page.detect_proxy_in_row(proxy_id)
```

不要把平台命令写进 `pages/*`。

### 3. 优先 CDP 和 DOM

文件上传优先级：

1. Playwright/CDP 设置 `input[type=file]`。
2. JS/DOM 找到隐藏 input 后触发。
3. 平台桌面文件选择器兜底。

截图优先级：

1. CDP 截图。
2. `mss` 桌面截图。
3. Windows UIAutomation 截图。
4. macOS/Linux 后续可选平台原生命令。

### 4. 平台能力缺失时 skip

如果当前平台缺少必要能力，应：

- 使用 `skipTest` 或框架统一 skip。
- 输出明确原因。
- 不要让用例因为 `ImportError`、命令不存在、权限缺失而失败。

示例：

```python
if not caps.desktop_file_dialog:
    self.skipTest("desktop file dialog fallback is not supported on this platform")
```

### 5. 依赖声明

后续 `requirements.txt` 应改成平台条件依赖：

```text
PyYAML>=6.0.1,<7.0
requests>=2.31.0,<3.0
playwright>=1.40.0,<2.0
websocket-client>=1.7.0,<2.0
openpyxl>=3.1.2,<4.0
mss>=9.0.1,<11.0
streamlit>=1.35.0,<2.0
psutil>=5.9.0,<6.0
uiautomation>=2.0.18,<3.0; platform_system == "Windows"
pywin32>=306; platform_system == "Windows"
```

macOS/Linux 系统依赖通过文档说明，不强行塞进 Python requirements。

## APP 生命周期平台配置

当前代码已接入 `core/app_config.py` 作为 APP 配置解析入口。旧配置仍然有效：

```yaml

  windows:
    app:
      executable_path: "C:/Program Files/Dicloak/Dicloak.exe"
      work_dir: "C:/Program Files/Dicloak"
      process_name: "DICloak.exe"
      browser_process_name: "GinsBrowser.exe"
  macos:
    app:
      executable_path: "/Applications/DICloak.app/Contents/MacOS/DICloak"
      process_name: "DICloak"
      browser_process_name: "GinsBrowser"
  linux:
    app:
      executable_path: "/opt/dicloak/dicloak"
      work_dir: "/opt/dicloak"
      process_name: "dicloak"
      browser_process_name: "ginsbrowser"
```

解析规则：

- 当前平台段存在时优先读取当前平台段。
- `executable_path` 和 `exe_path` 等价，建议新配置统一使用 `executable_path`。

- 当前平台段缺失字段时回退到旧 `app` 字段，便于逐步迁移。
- `browser_process_name` 用于内核进程发现、内核缓存可执行文件等待和抓包阻断类用例。
- 能力矩阵中 macOS/Linux 已标记支持进程命令行读取和监听端口读取；APP 托管启动仍需真实平台补验。

## 托管启动前端就绪等待

托管启动模式不应只依赖进程存在来判断 APP 可用。当前 runner 在启动 APP 后执行：

1. `AppManager.launch_fresh()`：结束旧 APP 并按配置启动新 APP。
2. `CDPDriver.connect()`：等待 CDP 端口可连接。
3. `CDPDriver.wait_for_app_ready()`：等待 Electron 前端从加载页进入登录页或主界面。
4. 释放 startup check 连接，再进入 unittest 执行。

`wait_for_app_ready()` 的 ready 条件：

- 页面 body 文本不再包含 `正在加载中` 或 `Loading`。
- 页面出现登录控件、主界面菜单或至少一类可操作控件。
- 默认超时读取 `app.startup_timeout`，当前配置为 60 秒。

验收命令：

```powershell
python run.py --config config/config.yaml --module environment_group_management
```

当前 Windows 验收结果：

```text
total=6 passed=6 failed=0 errors=0 skipped=0 flaky=1
```

该命令会在结束后关闭由框架启动的 DICloak APP。
