# 飞书通知接入说明

本文档说明 `api_test_framework` 项目中飞书通知的实现方式，方便开发者或 AI 快速复用、排查和扩展通知能力。

## 功能概览

飞书通知由项目根目录下的 `notify_feishu.py` 实现。自动化运行完成后，脚本会读取 Allure 结果文件，统计用例总数、通过数、失败数、异常数、跳过数和通过率，并通过飞书自定义机器人 Webhook 发送一条中文文本消息。

当前通知内容格式如下：

```text
Local API自动化已完成@指定人员
总用例：128
通过率：99.2%
通过：127
失败：0
异常：0
跳过：1
测试报告链接：http://192.168.20.33:9091/test_report.html
```

## 相关文件

- `notify_feishu.py`：飞书通知主脚本，负责读取配置、统计结果、组装消息、发送 Webhook。
- `config/env.yaml`：通知配置来源，包含 Webhook、@ 人信息、报告端口和报告文件名。
- `run.bat`：Windows 一键运行脚本，测试和报告生成后调用 `notify_feishu.py`。
- `run.sh`：Linux 一键运行脚本，测试和报告生成后调用 `notify_feishu.py`。
- `run_env_runtime_tests.bat`：长耗时环境打开/关闭场景脚本，也会调用 `notify_feishu.py`。
- `reports/allure-results`：默认 Allure 结果目录，通知统计从这里读取 `*-result.json`。
- `reports/test_report.html`：默认报告文件，通知消息中的报告链接指向该文件。

## 配置说明

通知配置位于 `config/env.yaml` 的 `notification` 节点：

```yaml
notification:
  # 飞书机器人 Webhook 地址。为空时跳过通知，不影响测试结果。
  webhook_url: "https://open.feishu.cn/open-apis/bot/v2/hook/xxxx"
  # 需要 @ 的飞书 open_id。为空时不 @ 任何人。
  at_open_id: "ou_xxxxx"
  # 通知中展示的联系人名称。需要和 at_open_id 同时配置才会生成 @ 标签。
  at_name: "马建勋"
```

报告链接依赖 `report` 节点：

```yaml
report:
  # Allure 原始结果目录，notify_feishu.py 会读取其中的 *-result.json。
  results_dir: "reports/allure-results"
  # HTML 报告文件名，用于拼接通知里的测试报告链接。
  html_file: "test_report.html"
  # 报告服务端口，用于拼接 http://本机IP:端口/报告文件名。
  port: 9091
```

注意：`webhook_url` 属于敏感信息。对外共享代码或文档时，建议替换为占位符，不要泄露真实机器人地址。

## 执行流程

一键运行脚本的调用顺序如下：

1. 执行 `run_tests.py` 运行 pytest 用例。
2. 执行 `generate_report.py` 生成 HTML 测试报告。
3. 执行 `serve_report.py` 启动报告服务。
4. 执行 `notify_feishu.py` 发送飞书通知。
5. 脚本最终使用 pytest 的退出码作为一键运行脚本退出码。

也可以单独发送通知：

```bash
python notify_feishu.py
```

单独执行时必须确保 `reports/allure-results` 中已有本次测试生成的 `*-result.json`，否则脚本会输出 `No test results found, skip notification` 并跳过发送。

## 统计逻辑

`notify_feishu.py` 的 `load_stats(results_dir)` 会扫描：

```text
reports/allure-results/*-result.json
```

并按 Allure 结果中的 `status` 字段统计：

- `passed`：通过
- `failed`：失败
- `broken`：异常
- `skipped`：跳过

通过率计算方式：

```text
通过率 = passed / total * 100
```

其中 `total` 是 `*-result.json` 文件数量，也就是 Allure 记录到的测试用例数量。

## 报告链接生成逻辑

脚本通过 `socket` 获取当前机器局域网 IP，并拼接报告链接：

```python
report_url = f"http://{local_ip}:{port}/{report_file}"
```

例如：

```text
http://192.168.20.33:9091/test_report.html
```

如果获取本机 IP 失败，会回退为：

```text
127.0.0.1
```

如果飞书群成员无法打开报告链接，优先检查：

- 运行机器和查看报告的人是否在同一局域网。
- `report.port` 是否和报告服务实际端口一致。
- Windows 防火墙或 Linux 防火墙是否放行该端口。
- `serve_report.py` 是否仍在运行。

## 飞书消息格式

当前使用飞书自定义机器人的文本消息格式：

```python
payload = {
    "msg_type": "text",
    "content": {
        "text": text,
    },
}
```

请求头：

```python
headers = {"Content-Type": "application/json; charset=utf-8"}
```

请求超时时间：

```python
timeout=10
```

@ 人格式：

```python
mention = f'<at user_id="{at_open_id}">{at_name}</at>'
```

只有同时配置 `at_open_id` 和 `at_name` 时，消息标题中才会包含 @ 标签。

## 成功和失败判断

发送成功条件：

```python
resp.status_code == 200 and resp.json().get("code") == 0
```

成功时输出：

```text
Feishu notification sent
```

以下情况会返回失败退出码 `1`：

- 网络请求异常，例如超时、连接失败、DNS 失败。
- 飞书返回 HTTP 非 200。
- 飞书返回 HTTP 200，但响应 JSON 中 `code != 0`。

以下情况会正常跳过并返回退出码 `0`：

- `notification.webhook_url` 为空。
- `reports/allure-results` 中没有 `*-result.json`。

## 快速接入步骤

1. 在飞书群中添加自定义机器人，复制 Webhook 地址。
2. 将 Webhook 填入 `config/env.yaml` 的 `notification.webhook_url`。
3. 如需 @ 指定人员，填入该人员的 `open_id` 和展示名称。
4. 确认 `report.results_dir`、`report.html_file`、`report.port` 与实际报告配置一致。
5. 使用 `run.bat` 或 `run.sh` 运行完整自动化流程。
6. 在飞书群中确认收到通知，并点击报告链接验证能打开。

## 常见问题

### 没有发送通知

优先检查：

- `notification.webhook_url` 是否为空。
- `reports/allure-results` 是否存在 `*-result.json`。
- 是否实际执行到了 `notify_feishu.py`。
- 控制台是否输出 `Webhook URL is empty, skip notification` 或 `No test results found, skip notification`。

### 飞书返回 code 非 0

常见原因：

- Webhook 地址错误或机器人已删除。
- 飞书机器人启用了关键词校验，但消息内容不包含关键词。
- 飞书机器人启用了签名校验，但当前脚本未实现签名逻辑。
- 飞书机器人安全策略限制了请求来源。

### @ 人不生效

优先检查：

- `at_open_id` 是否是飞书用户的 `open_id`，不是手机号、邮箱或昵称。
- `at_open_id` 和 `at_name` 是否同时配置。
- 被 @ 的人是否在接收消息的飞书群内。

### 报告链接打不开

优先检查：

- 报告服务是否启动。
- `report.port` 是否正确。
- 防火墙是否放行端口。
- 查看报告的人是否能访问运行机器的局域网 IP。

## 扩展建议

如果后续需要增强通知能力，可以考虑：

- 增加飞书签名校验支持，适配开启签名的机器人。
- 支持富文本或交互卡片，让报告链接更醒目。
- 在消息中追加失败用例名称摘要。
- 在配置中增加通知开关，例如 `notification.enabled`。
- 将 Webhook 改为从环境变量读取，降低敏感信息泄露风险。
